# 项目协作规则

本文件作用域：当前仓库根目录及其所有子目录。

## 沟通规则

### 修复 Bug 时禁止发散
- 当用户当前目标是“修复 bug / 修复报错 / 排查问题”时，优先只围绕当前问题作答。
- 回答顺序应尽量保持为：
  1. 根因
  2. 修复点
  3. 验证结果
- 在当前问题没有明确闭环前，不要主动发散到无关建议、后续优化、扩展功能或下一阶段规划。

### 只有在主问题闭环后，才能补充后续建议
- 如果需要补充“下一步可以做什么”，必须放在回答最后。
- 且补充内容应简短，不能盖过当前问题本身。

### 用户最后一句优先级最高
- 当用户最后一句是在追问某个具体 bug、报错、异常或修复结果时，不要切换到其它话题。
- 若用户要求“先不要开发，只讲方案”，则不得直接开始实现。

## 执行规则

### 先解决当前问题，再讨论扩展
- 编码时如果当前任务是修 bug，就先完成修复、验证、汇报。
- 不要在 bug 修复未确认完成前，顺带推进无关功能。

### 汇报必须给证据
- 修复类任务完成后，尽量明确说明：
  - 改动位置
  - 验证方式
  - 验证结果
