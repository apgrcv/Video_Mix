# 视频混剪合并工具 (Video_Mix) 项目全貌文档

> 更新时间：2026-03-17

---

## 1. 项目概述

本项目是一个基于 Python + Tkinter 的桌面端批量视频拼接工具，核心能力是通过 FFmpeg 将多个视频按规则批量拼接，生成大量成品视频。

当前产品包含两种任务模式：

1. **前中后拼接**
   - 使用“前半段目录”“中段目录（可选）”“后半段目录”三组素材生成结果。
   - 中段为空时，自动退化为前后两段拼接。
   - 输出数量为笛卡尔积：`前 × 后` 或 `前 × 中 × 后`。

2. **随机排列组合**
   - 使用一个统一素材目录。
   - 用户选择每个结果由 `2 / 3 / 4` 个视频组成。
   - 支持“拼接顺序”两种语义：
     - `顺序不同算不同结果`：输出数量为 `P(n, k)`。
     - `顺序不同算同一结果`：输出数量为 `C(n, k)`。
   - 在“算同一结果”模式下，仅片段集合相同、顺序不同时不会重复生成。

另外，当前版本支持**可选转场能力**：
- 可以在视频拼接点之间增加转场
- 支持开关控制
- 当前提供轻量时长配置（0.4 / 0.6 / 0.8 / 1.0 秒）

主要使用场景：短剧推广、素材批量混剪、不同片段顺序测试、批量生成多种组合视频。

## 2. 当前产品功能

### 2.1 界面结构

界面分为四块：

1. **顶部标题区**
   - 显示产品标题和一句功能说明。

2. **素材与输出区**
   - 顶部有两个 Tab：
     - `前中后拼接`
     - `随机排列组合`
   - 两种模式共用“输出目录”。
   - 鼠标滚轮滚动页面时不会触发 Tab 切换，避免误操作。
   - 所有“选择目录”按钮都会记住上一次选择的位置，并在下次打开系统弹框时自动定位到最近目录。

3. **合并选项区**
   - 合并模式
   - 统一分辨率
   - 跳过已存在文件
   - 转场效果
   - 并发数

4. **执行监控区**
   - 总任务进度
   - 剩余时间估算 ETA
   - 按线程拆分的日志面板

### 2.2 前中后拼接模式

输入项：
- 前半段目录
- 中段目录（可选）
- 后半段目录
- 输出目录

统计逻辑：
- 实时显示每个目录中的有效视频数量。
- 实时显示预计输出数量。

任务生成逻辑：
- 无中段时：生成 `(front, back)` 任务。
- 有中段时：生成 `(front, middle, back)` 任务。
- 输出文件名格式：`素材名_素材名[_素材名]_merged.mp4`

### 2.3 随机排列组合模式

输入项：
- 文件目录
- 拼接数量（2 / 3 / 4）
- 拼接顺序（顺序不同算不同结果 / 顺序不同算同一结果）
- 输出目录

统计逻辑：
- 实时显示文件目录中的有效视频数量。
- 实时显示当前拼接数量。
- 实时显示预计输出数量：
  - 顺序不同算不同结果：`P(n, k)`
  - 顺序不同算同一结果：`C(n, k)`

任务生成逻辑：
- 顺序不同算不同结果：`itertools.permutations(videos, k)`。
- 顺序不同算同一结果：`itertools.combinations(videos, k)`。
- 当目录中视频数少于拼接数量时，不允许开始任务。
- 输出文件名按参与拼接的顺序拼接，例如：`a_b_c_merged.mp4`。

说明：
- 当前命名虽然叫“随机排列组合”，实现不是随机抽样，而是穷举式生成：
  - 区分顺序时穷举所有排列
  - 不区分顺序时穷举所有组合

### 2.3.1 目录记忆

当前版本已支持目录选择记忆：
- 每个目录字段都会记住自己上一次成功选择的目录
- 如果某个字段没有历史记录，则回退到“最近一次浏览过的目录”
- 软件关闭后再次打开，仍然会保留这些目录记忆
- 本地记忆文件保存在项目根目录：`.video_mix_settings.json`

### 2.4 转场效果

新增可选转场功能，位于“合并选项”区域。

配置项：
- **启用转场效果**：默认关闭
- **轻量转场时长**（启用后显示）：
  - 0.4 秒
  - 0.6 秒
  - 0.8 秒
  - 1.0 秒

当前行为：
- 关闭时：走当前原有拼接逻辑
- 开启时：在每个拼接点之间插入转场
- 轻量转场：在片段之间插入极短黑场过渡，效果比直接切换更自然；时长可选。

### 2.5 合并模式

#### 1. 优先无损（失败自动转兼容）
- 在未启用转场时，先尝试无损拼接。
- 条件是参与拼接的视频流参数完全一致。
- 无损失败时自动回退到兼容模式。
- 无损模式使用 `-fflags +genpts`，用于修复时间戳兼容问题。

#### 2. 强制兼容（统一分辨率）
- 无论源视频参数是否一致，都直接重编码。
- 适用于分辨率、编码、帧率不一致的素材。

说明：
- **启用转场后，会自动走兼容重编码分支**，不再尝试无损拼接。

### 2.6 分辨率策略

#### 自动
- 以前当前任务中的**第一个视频**分辨率为基准。
- 在前中后模式里，通常就是前半段视频的分辨率。
- 在随机排列模式里，取当前排列结果中的第一个视频分辨率。

#### 自定义
- 用户手动输入目标宽和高，例如 `1080 x 1920`。

#### 画面处理方式
- 普通兼容拼接使用：`scale + pad + setsar=1 + format=yuv420p`
- 转场拼接使用：`scale + pad + fps=30 + settb=AVTB + setsar=1 + format=yuv420p`
- 目标是保持比例、不拉伸，并提升转场时不同素材之间的兼容性。

### 2.7 音频策略
- 所有输入片段都有音频时：拼接音频和视频。
- 只要任意片段缺少音频：输出纯视频结果。
- 启用转场时，会对片段边界音频做淡入淡出处理；轻量模式的黑场片段附带短静音音轨。

### 2.8 并发与日志
- 使用 `ThreadPoolExecutor` 做并发执行。
- 并发数由用户设置，默认取一个保守值（2 到 4）。
- 日志区域会根据并发数自动拆分成多个线程面板。
- 每个面板显示：
  - 当前处理文件
  - 当前线程日志
  - ETA 倒计时
- 启用转场时，日志会显示当前使用的转场类型和转场时长。
- 点击“停止任务”时，会终止正在运行的 FFmpeg 进程，并尝试删除未完成输出文件。

### 2.9 跨平台能力
- **macOS**：
  - 普通兼容拼接优先使用 `h264_videotoolbox`
  - 启用转场时，当前实现会改用 `libx264` 提高稳定性
- **Windows / Linux**：使用 `libx264`，参数为 `veryfast + crf 23`
- 打开输出目录时自动适配 Finder / Explorer / xdg-open。

## 3. 技术实现要点

### 3.1 技术栈
- 语言：Python 3.10+
- GUI：Tkinter + ttk.Notebook
- 视频处理：FFmpeg / FFprobe
- 并发：`ThreadPoolExecutor`
- 路径处理：`pathlib.Path`

### 3.2 当前核心文件
- `main.py`
  - GUI 主程序
  - 模式切换
  - 目录统计
  - 任务生成
  - 转场配置与转场拼接
  - 并发调度
  - FFmpeg 调用
- `cli.py`
  - 早期命令行版本
  - 当前能力少于 GUI，尚未同步最新双模式和转场逻辑

### 3.3 拼接内核
当前主程序已经统一支持 2 / 3 / 4 段拼接：
- 无损拼接：`concat_copy_many()`
- 普通兼容拼接：`concat_reencode_many()`
- 轻量转场：先对每个源素材标准化一次，再只处理头尾小片段，最后使用 `concat_copy_many()` 快速拼接

### 3.4 转场实现要点
- 轻量转场：对片段首尾的小窗口做 `fade/afade` 预处理，并插入极短黑场 + 静音过渡片段；主体部分尽量走快速 copy，减少重复编码
- 两种模式都会先把片段标准化，再尽量走快速 concat 复制链

### 3.5 任务生成流程
1. 根据当前 Tab 选择任务模式。
2. 扫描目录，筛选有效视频文件。
3. 根据模式生成待执行任务：
   - 前中后模式：笛卡尔积
   - 随机排列组合模式：按“拼接顺序”配置走全排列或全组合
4. 若勾选“跳过已存在文件”，则先过滤掉已存在输出。
5. 若启用了转场：
   - 先为当前输出确定转场类型
   - 然后走兼容重编码 + 转场滤镜链
6. 把任务提交给线程池执行。
7. 实时更新完成数、日志和 ETA。

## 4. 当前实现边界

- `随机排列组合` 不是随机抽样，而是穷举生成（排列或组合，取决于“拼接顺序”配置）。
- 启用转场后仍会比无损拼接慢，但相较旧的完整滤镜重叠方案，速度已大幅提升。
- 当片段过短时，转场时长会被自动压缩；如果仍然太短，则任务会失败并记录日志。
- 输出文件名使用素材文件名 `stem` 拼接，如果不同文件仅扩展名不同且主名相同，存在文件名冲突风险。
- `cli.py` 仍然是旧逻辑，不代表 GUI 的完整能力。
- 当前 UI 仍基于原生 Tk/ttk 外观，没有引入额外主题库。
