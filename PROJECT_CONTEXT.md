## 1. 项目概述

本项目是一个用于**批量视频合并**的桌面端工具（基于 Python + Tkinter）。
核心功能是将“前半段视频素材”与“后半段视频素材”进行**笛卡尔积（全排列）组合**，自动生成所有可能的合并视频。

**主要场景**：短剧推广、视频混剪，需要将多个开头与多个结尾批量拼接。

---

## 2. 核心业务功能

### 2.1 视频输入与输出
- **前半段目录**：选择存放前半段视频的文件夹。
- **后半段目录**：选择存放后半段视频的文件夹。
- **输出目录**：指定合并后视频的保存位置。
- **智能统计**：
  - 自动显示前半段、后半段目录下的有效视频数量。
  - **预计输出数量**：实时计算并显示 `前半段数量 × 后半段数量`，让用户对任务规模有预期。

### 2.2 合并模式
工具提供两种核心合并模式：

1.  **开头目录（新增）**
    - **用途**：在"前×中×后"拼接的基础上，在开头添加一个视频片段
    - **拼接方式**：顺序拼接（不是笛卡尔积相乘）
    - **示例**：
      - 假设 "前×中×后" 生成 90 个视频组合
      - 开头目录有 9 个视频
      - 最终结果：90 个视频，每个视频前面都按顺序添加一个开头视频
      - 第1个组合 + 开头第1个视频
      - 第2个组合 + 开头第2个视频
      - ...
      - 第10个组合 + 开头第1个视频（循环使用）
    - **技术实现**：
      - 使用 `pick_intro()` 方法选择开头目录
      - 在 `run_merge()` 中遍历组合时，按顺序取用开头视频
      - 使用取模运算 `intro_index % len(intros)` 实现循环使用

2.  **前×中×后拼接**
    - 支持"前半段"、"中间段"、"后半段"三个视频段落的任意组合
    - 中间段为可选（非必填）
    - 生成笛卡尔积：前半段数 × 中间段数 × 后半段数 = 输出数量

3.  **优先无损模式（推荐）**
    - **逻辑**：首先尝试“无损拼接”（Copy Stream），速度极快，不损耗画质。
    - **条件**：要求前后两段视频的编码格式、分辨率、时基等参数完全一致。
    - **自动回退**：如果无损拼接失败（或参数不兼容），自动降级为“兼容模式”进行重编码合并。
    - **关键修复**：针对无损模式下“后半段黑屏（有声音无画面）”的问题，已通过添加 FFmpeg `-fflags +genpts` 参数解决，强制重建时间戳。

2.  **强制兼容模式（统一分辨率）**
    - **逻辑**：无论源视频参数如何，强制对所有视频进行重编码。
    - **用途**：解决源视频分辨率不一、编码混乱导致的合并失败问题。
    - **分辨率策略**：
        - **自动**：以前半段视频的分辨率为基准。
        - **自定义**：用户手动输入目标宽、高（例如 1080x1920）。
    - **画面处理**：使用 `scale` + `pad` 滤镜，保持原始比例，背景填充黑边，防止画面拉伸变形。

### 2.3 性能与并发
- **多线程支持**：用户可设置"并发数"（线程数），默认为 CPU 核心数的一半（2-4）。
- **跨平台性能优化**：
    - **macOS**：优先调用 Apple Silicon / Intel 核显的 **`h264_videotoolbox`** 硬件编码器。
        - **参数**：码率 `10000k` (10Mbps)，`allow_sw 1`。
        - **效果**：速度提升 5-10 倍，CPU 占用极低。
    - **Windows/Linux**：使用 `libx264` 软编码。
        - **参数**：Preset `veryfast`，CRF `23`（平衡速度与画质）。
        - **优化原因**：Windows 下硬件加速支持不如 macOS 完善，因此采用经过优化的 x264 编码参数。

### 2.4 进度与日志监控
- **总进度条**：显示 `已完成 / 总任务数`。
- **独立线程日志（UI 亮点）**：
    - 日志区域根据并发数**自动拆分为多个独立方块**（Grid 布局）。
    - 每个方块对应一个后台线程，独立显示该线程当前的日志流，互不干扰。
- **精准倒计时（ETA）**：
    - 解析 FFmpeg 输出的 `out_time_ms`，实时计算当前任务的剩余时间。
    - 每个线程方块顶部独立显示 `ETA: MM:SS`。

### 2.5 跨平台兼容性 (macOS & Windows)
- **业务逻辑**：
    - 软件设计之初即考虑到多端分发，确保用户无论使用 MacBook 还是 Windows 电脑，都能获得一致的操作体验和核心功能。
    - **路径处理**：自动识别系统路径分隔符（`/` vs `\`），用户无需关心底层差异。
    - **文件打开**：合并完成后，点击“打开输出目录”按钮，软件会根据操作系统自动调用 Finder (macOS) 或 文件资源管理器 (Windows)。
    - **性能适配**：针对不同系统的优势进行差异化调度（如 macOS 使用 Metal/VideoToolbox 加速，Windows 使用兼容性更强的 x264）。

## 3. 技术实现细节

### 3.1 架构栈
- **语言**：Python 3.10+
    - **跨平台基石**：Python 解释器本身屏蔽了大部分 OS 差异。
    - **路径库**：全线使用 `pathlib.Path`，它会自动根据 OS 处理路径分隔符，避免了硬编码 `\` 或 `/` 导致的兼容性 Bug。
- **GUI 框架**：Tkinter（原生库，轻量级）。
    - Tkinter 调用底层 Tcl/Tk，自动渲染为当前操作系统的原生窗口风格（macOS 的 Aqua 风格，Windows 的 Aero/Metro 风格）。
- **核心依赖**：FFmpeg (命令行工具)，通过 `subprocess` 调用。
    - **动态二进制查找**：
        - 程序启动时会遍历当前目录及 `ffmpeg/` 子目录。
        - 自动判断后缀：Windows 下寻找 `ffmpeg.exe`，macOS/Linux 下寻找 `ffmpeg`。
        - 优先使用内置打包的二进制文件，若无则尝试调用系统环境变量中的 `ffmpeg`。
- **打包**：支持 PyInstaller 打包为 .app 或 .exe。
    - **macOS**：打包为 `.app` 或 Unix 可执行文件，包含 `ffmpeg` 二进制。
    - **Windows**：打包为 `.exe`，包含 `ffmpeg.exe`。

### 3.2 关键代码逻辑 (`main.py`)

#### A. 跨平台系统调用
- **打开文件夹**：
  ```python
  if os.name == "nt":
      os.startfile(out_dir)  # Windows 专用 API
  elif sys.platform == "darwin":
      subprocess.run(["open", out_dir])  # macOS `open` 命令
  else:
      subprocess.run(["xdg-open", out_dir])  # Linux 标准
  ```
- **硬件加速判定**：
  ```python
  use_videotoolbox = sys.platform == "darwin"  # 仅在 macOS 启用 VideoToolbox
  ```

#### B. 无损拼接逻辑 (`concat_copy`)
- **FFmpeg 命令**：
  ```bash
  ffmpeg -f concat -safe 0 -i list.txt -c copy -fflags +genpts output.mp4
  ```
- **核心点**：`+genpts` 是解决时间戳错乱的关键。

#### B. 重编码逻辑 (`concat_reencode`)
- **滤镜链 (`filter_complex`)**：
  ```
  [0:v]scale=...:force_original_aspect_ratio=decrease,pad=...:setsar=1,format=yuv420p[v0];
  [1:v]scale=...[v1];
  [v0][v1]concat=n=2:v=1:a=1[v][a]
  ```
- **关键参数**：
  - `setsar=1`：强制设置像素比为 1:1，防止个别播放器显示变形。
  - `format=yuv420p`：确保兼容性（避免 yuv444 等生僻格式）。

#### C. 多线程与 UI 通信
- **线程模型**：使用 `concurrent.futures.ThreadPoolExecutor`。
- **线程槽位 (`thread_slots`)**：
  - 为了将线程与 UI 上的日志方块固定绑定，实现了一个 `Queue` 管理的槽位机制。
  - 线程启动前领取一个 `slot_id` (0, 1, 2...)，结束后归还。
  - UI 根据 `slot_id` 将日志更新到对应的 `ScrolledText` 组件。
- **消息队列 (`queue.Queue`)**：
  - 子线程不直接操作 UI（防止崩溃）。
  - 子线程发送 `(action, data)` 到队列。
  - 主线程通过 `root.after` 定时轮询队列并更新 UI。

## 4. 历史修复与优化记录

1.  **UI 假死与卡顿**
    - **问题**：并发数 `IntVar` 获取值时未做类型转换，导致线程池初始化失败。
    - **修复**：强制 `int()` 转换并增加异常捕获。

2.  **无损模式后半段黑屏**
    - **问题**：直接拼接不同来源的视频，时间戳（PTS/DTS）不连续。
    - **修复**：引入 `-fflags +genpts` 重建索引。

3.  **导出速度慢**
    - **问题**：默认使用 `libx264` 软编码，且预设为 `medium`，效率低。
    - **修复**：macOS启用 `videotoolbox` 硬编；软编预设改为 `veryfast`。

4.  **日志混乱**
    - **问题**：多线程日志全部打印到一个文本框，无法分清哪个线程在跑哪个任务。
    - **修复**：实现了动态 Grid 布局的独立日志面板。

5.  **开头目录顺序拼接（新增）**
    - **问题**：用户需要在"前×中×后"拼接的基础上添加开头视频。
    - **需求**：顺序拼接（不是相乘），例如 90 个组合 + 9 个开头视频 = 90 个最终视频。
    - **修复**：新增 `pick_intro()` 方法和 `intro_dir` 变量，在 `run_merge()` 中实现顺序取用逻辑。
