> 更新时间：2026-03-08

---

## 1. 项目概述

本项目是一个基于 Python + Tkinter 的桌面端批量视频拼接工具，核心能力是通过 FFmpeg 将多个视频片段按规则批量拼接，输出大量成品视频。

当前产品提供两种任务模式：

1. 前中后拼接
   - 使用“前半段”“中段（可选）”“后半段”三组素材生成结果。
   - 中段为空时，退化为前后两段拼接。
   - 结果数量为笛卡尔积：`前 × 后` 或 `前 × 中 × 后`。

2. 随机排列组合
   - 使用一个统一素材目录。
   - 用户选择每个结果由 `2 / 3 / 4` 个视频组成。
   - 系统对目录内视频做全排列，结果数量为 `P(n, k)`。
   - 例如目录内有 4 个视频，拼接数量选择 3，则结果总数为 `P(4,3)=24`。

主要场景：短剧推广、混剪分发、不同素材顺序测试、批量内容组合生成。

## 2. 当前产品功能

### 2.1 界面结构
- 顶部提供两个 Tab：
  - `前中后拼接`：默认模式。
  - `随机排列组合`：新增模式。
- “开头目录（可选）”已从界面中移除，不再参与当前产品流程。
- `合并模式`、`统一分辨率`、`跳过已存在文件`、`任务进度`、`运行日志` 为两种 Tab 共用。

### 2.2 前中后拼接模式
- 输入项：
  - 前半段目录
  - 中段目录（可选）
  - 后半段目录
  - 输出目录
- 统计逻辑：
  - 实时显示各目录中的有效视频数量。
  - 实时显示预计输出数量。
- 任务生成逻辑：
  - 无中段时生成两段任务：`front + back`
  - 有中段时生成三段任务：`front + middle + back`

### 2.3 随机排列组合模式
- 输入项：
  - 文件目录
  - 拼接数量（2 / 3 / 4）
  - 输出目录
- 统计逻辑：
  - 实时显示文件目录中的有效视频数量。
  - 实时显示当前拼接数量。
  - 实时显示预计输出数量：`P(n, k)`。
- 任务生成逻辑：
  - 对目录内所有视频执行 `itertools.permutations(videos, k)`。
  - 顺序不同视为不同结果。
  - 输出文件名按拼接顺序拼接素材名，例如：`a_b_c_merged.mp4`。

### 2.4 合并模式

#### 优先无损（失败自动转兼容）
- 先尝试 FFmpeg concat demuxer + stream copy。
- 要求参与拼接的视频流参数完全一致。
- 无损失败时自动回退到兼容模式。
- 使用 `-fflags +genpts` 解决无损拼接后可能出现的时间戳问题。

#### 强制兼容（统一分辨率）
- 无论源素材参数是否一致，统一进行重编码。
- 用于处理编码格式、分辨率、帧率不一致导致的兼容问题。

### 2.5 分辨率策略
- 自动：以前半段或当前任务中的第一个视频分辨率为基准。
- 自定义：用户输入目标宽度和高度。
- 滤镜策略：`scale + pad + setsar=1 + format=yuv420p`
  - 保持原始比例
  - 黑边补齐画布
  - 防止拉伸、花屏、黑屏等兼容问题

### 2.6 音频策略
- 参与拼接的所有片段都有音频时：拼接音视频。
- 只要其中任意一个片段无音频时：输出纯视频结果。

### 2.7 并发与日志
- 支持设置并发数，默认取 CPU 核心数的一个保守值（2-4）。
- 使用 `ThreadPoolExecutor` 并发执行 FFmpeg 任务。
- 日志区域会根据并发数动态拆成多个线程面板。
- 每个线程面板独立显示：
  - 当前处理文件
  - 实时 FFmpeg 输出
  - ETA 倒计时
- 点击“停止任务”会终止当前 FFmpeg 进程，并尝试删除未完成输出文件。

## 3. 技术实现要点

### 3.1 技术栈
- 语言：Python 3.10+
- GUI：Tkinter + ttk.Notebook
- 视频处理：FFmpeg / FFprobe
- 并发：`concurrent.futures.ThreadPoolExecutor`
- 路径处理：`pathlib.Path`

### 3.2 核心文件
- `main.py`
  - GUI 主程序
  - 目录统计
  - 任务生成
  - 并发调度
  - FFmpeg 调用
- `cli.py`
  - 早期命令行版本
  - 当前能力明显少于 GUI，不代表最新产品形态
- `使用说明.md`
  - 面向使用者的操作说明
- `技术方案.md`
  - 面向迭代开发的简版技术说明

### 3.3 拼接内核
当前主程序已经统一支持 2 / 3 / 4 段拼接：
- 无损拼接：`concat_copy_many()`
- 兼容拼接：`concat_reencode_many()`

这样两种任务模式都可以复用同一套执行逻辑，后续若要继续扩展到更多段数，也可以沿用当前方向。

### 3.4 任务生成流程
1. 根据当前 Tab 决定任务模式。
2. 扫描目录，收集有效视频文件。
3. 根据模式生成待执行任务列表：
   - 前中后模式：笛卡尔积
   - 随机排列组合模式：全排列
4. 若勾选“跳过已存在文件”，则先过滤已存在结果。
5. 把任务提交给线程池执行。
6. 汇总完成数、日志和 ETA。

## 4. 跨平台行为
- macOS
  - 优先使用 `h264_videotoolbox` 做硬件编码
- Windows / Linux
  - 使用 `libx264`，参数为 `veryfast + crf 23`
- 输出目录打开逻辑按系统自动切换 Finder / Explorer / xdg-open

---

## 5. 当前已知边界
- `随机排列组合` 当前是“全排列生成”，名称里虽然有“随机”，但实际并不是随机抽样，而是穷举所有顺序结果。
- 输出文件名使用素材 `stem` 拼接，如果不同文件恰好同名不同扩展名，可能产生结果名冲突。
- `cli.py` 还没有同步支持新的双 Tab 业务模型，当前以 GUI 为准。
